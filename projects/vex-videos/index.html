Displaying Videos on a VEX V5 Brain

Custom image compression and a lightweight renderer for real-time animation on a constrained embedded system

2026-05-23

Intro

In 2026, I competed in the VEX V5 Robotics Competition with my team, the Imagineers. Between working on the robot and writing the autonomous code, I made this as a side project to keep myself busy and hopefully make our robot stand out.

My goal was to give the robot a face that could react dynamically during gameplay. We programmed our robot using the PROS C++ framework developed by Purdue University. The problem was that PROS has no built-in support for rendering images or videos. This meant I had to create a custom renderer from scratch, using the limited drawing functions available.

The biggest challenge was displaying images efficiently. Once I had that solved, playing videos was simply a matter of displaying images in sequence. My approach was to convert PNG files into vectors of values that the renderer could interpret and reconstruct directly on the Brain’s screen.

Converter

I wrote the conversion code in Python, as it's the language I'm most comfortable with. My initial solution was inspired by a tool developed by Suhjae, which converts PNG images into compressed C++ vectors for rendering on the V5 Brain.

I later rewrote the system to be more efficient and better suited for video playback. Instead of generating a unique colour palette for each frame, the new system uses a shared, pre-defined palette to reduce both memory usage and redundancy across frames. Images are then quantized down to 16 colours, mapping each pixel to the closest available palette entry. This keeps the representation compact while also reducing the number of colour changes needed during rendering.

The data stream combines run-length encoding with inline colour changes. Negative values indicate a switch in palette colour, while positive values represent a run of pixels to draw using the current colour. The renderer can also skip pixels that are unchanged from the previous frame, improving performance during animation.

Renderer

The renderer is written in C++ and uses the same fixed palette generated by the conversion tool. It reads the compressed data vector and reconstructs each frame at runtime.

Instead of drawing individual pixels, it draws horizontal runs and allows them to span multiple lines on the 480×240 screen. Runs that exceed a line are split automatically.

To improve performance, unchanged pixels from previous frames are skipped entirely by offsetting the draw position. The renderer runs at ~24 FPS (42 ms delay) in a separate thread so it does not interfere with robot control code, and is kept deliberately simple to minimise load on the Brain.

The Face

With the rendering system finished, the last step was making the face itself. The design was heavily inspired by Pudu Robotics’ BellaBot. The animations were made using Rive, and exported as PNG sequences before being processed by the conversion tool.

Once all the animations were on the brain, I implemented a simple state machine to change expressions based on controller input. When the robot was idle, the face automatically cycled through idle animations to make it feel more alive.
